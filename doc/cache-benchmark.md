# 缓存压测记录

## 测试目标

- 接口：`GET /app/room/getDetailById?id=<roomId>`
- 目标：对比房源详情热点接口在“关闭缓存”和“开启 Redis 热点缓存”两种场景下的性能差异
- 缓存开关：`lease.cache.hot-data-enabled`

## 测试配置

- 压测工具：JMeter
- 并发参数：`1000` 线程 / `5s` Ramp-up / `20` 次循环
- 总样本数：`20000`
- 压测环境：本地单机环境（应用、MySQL、Redis、JMeter 均在本机）

## 测试结果

| 场景 | 样本数 | Avg(ms) | Median(ms) | P90(ms) | P95(ms) | P99(ms) | Min(ms) | Max(ms) | Throughput(req/s) | Error % |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 无 Redis 缓存 | 20000 | 2381 | 2374 | 2864 | 3007 | 3212 | 94 | 3612 | 379.54 | 0.00% |
| 开启 Redis 缓存 | 20000 | 1018 | 1159 | 1357 | 1437 | 1562 | 12 | 1697 | 724.09 | 0.00% |

## 对比结论

| 指标 | 无 Redis | 有 Redis | 优化幅度 |
| --- | ---: | ---: | ---: |
| 平均响应时间 | 2381 ms | 1018 ms | 下降 57.2% |
| 中位数 | 2374 ms | 1159 ms | 下降 51.2% |
| P90 | 2864 ms | 1357 ms | 下降 52.6% |
| P95 | 3007 ms | 1437 ms | 下降 52.2% |
| P99 | 3212 ms | 1562 ms | 下降 51.4% |
| 最大响应时间 | 3612 ms | 1697 ms | 下降 53.0% |
| 吞吐量 | 379.54 req/s | 724.09 req/s | 提升 90.8% |
| 错误率 | 0.00% | 0.00% | 持平 |

## 结论说明

- 开启 Redis 热点缓存后，房源详情接口的平均响应时间从 `2381ms` 降至 `1018ms`，尾延迟也显著下降。
- 吞吐量从 `379.54 req/s` 提升到 `724.09 req/s`，在本地单机环境下已能体现出明显收益。
- 该结果可以作为“热点数据缓存有效”的基线对照；若要进一步验证 `Redisson` 分布式锁对缓存击穿的抑制效果，建议补充“删除热点 key 后瞬时并发冲击”的第三组压测。
