# Caffeine 两级缓存压测记录

## 1. 测试目标

- 验证在 `Redis` 热点缓存已经命中的前提下，引入 `Caffeine` 本地缓存后，热点详情接口是否还能继续降低响应时间并提升吞吐。
- 验证对象：`GET /app/apartment/getDetailById?id=9`

## 2. 测试说明

- 压测工具：`JMeter`
- 压测时长：`60s`
- 并发线程：`500`
- 测试环境：本地单机环境（应用、MySQL、Redis、JMeter 均在本机）
- 测试方式：先预热热点 key，再对同一热点详情接口进行压测

本次压测分为两组：

- `FALSE`：关闭 `Caffeine` 本地缓存，仅保留 `Redis` 热点缓存
- `TRUE`：开启 `Caffeine` 本地缓存，读链路为 `Caffeine -> Redis -> DB`

说明：

- `lease.cache.hot-data-enabled=true`
- 这里对比的核心变量是本地缓存开关，即 `lease.cache.local-enabled`
- 以下数据根据压测结果截图整理，使用 `Aggregate Report` 口径进行对比

## 3. 压测结果

| 场景 | Samples | Avg(ms) | Median(ms) | P90(ms) | P95(ms) | P99(ms) | Min(ms) | Max(ms) | Error % | Throughput(req/s) |

## 4. 对比结论

| 指标 | 关闭本地缓存 | 开启本地缓存 | 优化幅度 |
| --- | ---: | ---: | ---: |
| 平均响应时间 | 879 ms | 295 ms | 下降 66.4% |
| 中位数 | 883 ms | 288 ms | 下降 67.4% |
| P90 | 1170 ms | 398 ms | 下降 66.0% |
| P95 | 1283 ms | 435 ms | 下降 66.1% |
| P99 | 1663 ms | 514 ms | 下降 69.1% |
| 吞吐量 | 541.32 req/s | 1612.91 req/s | 提升 198.0% |
| 错误率 | 0.00% | 0.00% | 持平 |

结论：

- 在热点详情接口场景下，引入 `Caffeine + Redis` 两级缓存后，响应时间和吞吐都获得了非常明显的改善。
- 尤其是尾延迟指标改善更明显，`P99` 从 `1663ms` 降至 `514ms`，下降约 `69.1%`。
- 吞吐量从 `541.32 req/s` 提升到 `1612.91 req/s`，接近 `2.98 倍`。
- 在本次压测中，两组错误率均为 `0%`，说明优化没有带来额外稳定性问题。

## 5. 技术解读

这次收益主要来自以下几点：

- 热点详情数据在应用进程内直接命中 `Caffeine`，避免了每次请求都访问 `Redis`
- 降低了 `Redis` 网络往返和序列化开销
- 在高并发场景下，尾延迟收敛更加明显
- 保留了原有的空值缓存、TTL 抖动、`Redisson` 分布式锁逻辑，不影响原有缓存治理策略

当前方案边界：

- 该优化更适合 `读多写少` 的热点详情类接口
- 当前本地缓存失效已支持单进程场景下的主动淘汰
- 若是多实例部署，后续还可结合 `Redis Pub/Sub` 或消息广播继续完善跨实例本地缓存失效

## 6. 简历表述建议

可直接写成：

> 针对公寓详情热点接口引入 `Caffeine + Redis` 两级缓存，结合空值缓存、TTL 抖动与 `Redisson` 分布式锁优化读链路；在本地 `500` 线程、`60s` 压测下，平均响应时间由 `879ms` 降至 `295ms`，`P99` 由 `1663ms` 降至 `514ms`，吞吐由 `541 req/s` 提升至 `1613 req/s`。

## 7. 面试表述建议

可以这样回答：

> 在 Redis 热点缓存基础上，我又为详情接口补了一层 `Caffeine` 本地缓存，把读链路优化成 `Caffeine -> Redis -> DB`。这样做的目的不是替代 Redis，而是进一步降低热点请求对 Redis 的压力，并优化高并发下的尾延迟。  
> 我做了 `500` 线程、`60s` 的本地压测，对同一个热点公寓详情接口进行 A/B 对比。关闭本地缓存时平均响应时间是 `879ms`，开启后降到 `295ms`；`P99` 从 `1663ms` 降到 `514ms`；吞吐从 `541 req/s` 提升到 `1613 req/s`。  
> 同时我也补了本地缓存失效能力，后台更新房源和房间信息时会主动淘汰缓存。单实例场景已经闭环，多实例场景后续可以用 `Redis Pub/Sub` 做进一步完善。
